════════════════════════════════════════
LESSON STRUCTURE
════════════════════════════════════════

6 stages. Each stage has exactly 5 rounds.
Pick exactly 6 different mechanics — NO mechanic may appear twice in the lesson (each stage = one unique interaction type).

Stage 1 — always the gentlest entry point (must be drag_drop, tap_count, OR drag_sort only).
Stages 2–5 — escalating complexity
Stage 6 — hardest challenge of the lesson + finale feel

════════════════════════════════════════
MECHANIC SELECTION
════════════════════════════════════════

Pick exactly 6 different mechanics from this list.
Never use the same mechanic twice in one lesson.

Available mechanics:
  drag_drop       — drag items into target zones
  drag_sort       — arrange elements in correct order
  drag_group      — sort into two named groups
  pattern_input   — fill missing number in a sequence
  fill_blank      — solve and type one answer
  multi_choice    — pick one correct answer from three
  corridor_choice — choose left or right path
  match_pairs     — find two correct pairs from four options
  tap_count       — tap a target exactly N times
  balance_scale   — make both sides of equation equal
  build_number    — compose a number from parts
  timer_challenge — solve 3 examples within time limit
  symbol_calc     — compute expression where A/B/C = numbers
  find_unknown    — find missing variable in equation

STAGE 1 RULE:
  Must always be drag_drop, tap_count, OR drag_sort.
  This is the warmup — confidence before challenge.

AGE RULES:
  age 5–6 → only: drag_drop, tap_count, drag_sort,
             multi_choice, fill_blank, drag_group
  age 7–8 → all except timer_challenge (use rarely)
  age 9   → all including timer_challenge

DIFFICULTY RULES (parsed from tutor_prompt):
  level 1–3 → prefer: drag_drop, tap_count, drag_sort,
               multi_choice, fill_blank
               numbers: age 5–6 up to 10, age 7–9 up to 20

  level 4–6 → add: pattern_input, corridor_choice,
               balance_scale, drag_group
               numbers: age 5–6 up to 10, age 7–9 up to 50

  level 7–10 → add: symbol_calc, find_unknown,
                timer_challenge, match_pairs
                numbers: up to 100, complex expressions allowed

════════════════════════════════════════
MECHANIC FIELD REFERENCE — what each mechanic needs in JSON
════════════════════════════════════════

Every example object MUST include: "roundNumber" (1–5), "titleText" (short stage card title for that round).

Then add ONLY the fields listed for that stage's mechanic. Types: strings unless noted as number or array.

--- drag_drop ---
Game: child drags item tokens into bottom target zones; each zone needs a fixed count of correct drops.
Per round REQUIRED fields:
  • screenItemCount (number) — total draggable items on screen
  • targetCount (number) — how many drop zones (usually 2)
  • zoneCounts (array of numbers, length = targetCount) — how many correct items belong in each zone; sum should equal screenItemCount for a clean “share equally” puzzle, or your intentional split
Optional: prompt if you use a one-line story; keep titleText as the main visible title.

--- drag_sort ---
Game: numbers appear as tokens; child orders them (e.g. ascending).
Per round REQUIRED fields:
  • prompt (string) — comma/space separated values shown as tokens, e.g. "2, 5, 3, 8, 1"
  • answer (string) — rule or ordered list the engine expects, e.g. "ascending" or "1,2,3,5,8"

--- drag_group ---
Game: child sorts listed numbers into two named groups (e.g. even / odd).
Per round REQUIRED fields:
  • numbersText (string) — all numbers that appear, e.g. "2, 3, 6, 9, 4, 8"
  • group1Name, group2Name (strings) — labels for the two bins
  • group1Values, group2Values (strings) — comma-separated correct values per group; every number in numbersText must appear in exactly one group

--- pattern_input ---
Game: child types the missing term in a visible sequence.
Per round REQUIRED fields:
  • prompt (string) — sequence with missing slot, e.g. "2, 4, 6, ?"
  • answer (string) — correct missing value

--- fill_blank ---
Game: one equation or phrase with a blank; child types the answer.
Per round REQUIRED fields:
  • prompt (string) — the problem text including the blank, e.g. "12 ÷ 3 ="
  • answer (string) — exact correct value (digits only unless you intentionally use words)

--- multi_choice ---
Game: three tap options A/B/C; exactly one correct.
Per round REQUIRED fields:
  • prompt (string) — task text in the play area
  • choiceA, choiceB, choiceC (strings) — option labels
  • correctOption (string) — exactly one of: "A", "B", "C"

--- corridor_choice ---
Game: child picks left or right path comparing two math expressions.
Per round REQUIRED fields:
  • prompt (string) — short story question, e.g. "Which side is bigger?"
  • leftExpression, rightExpression (strings) — math only, e.g. "3 × 4" vs "2 × 5"
  • correctSide (string) — exactly "left" or "right"

--- match_pairs ---
Game: four statements A–D; exactly two are correct; child finds both.
Per round REQUIRED fields:
  • pairA, pairB, pairC, pairD (strings) — four lines (can be equations with =)
  • correctPair1, correctPair2 (strings) — each is one of "A","B","C","D"; must be two DIFFERENT letters for the two true statements

--- tap_count ---
Game: child taps one on-screen item exactly N times.
Per round REQUIRED fields:
  • prompt (string) — instruction, e.g. "Tap the banana 6 times"
  • answer (string or number) — required tap count as digits, e.g. "6"

--- balance_scale ---
Game: left bowl has an expression with ?; child types what replaces ? to balance the right side.
Per round REQUIRED fields:
  • leftExpression (string) — include "?" for unknown, e.g. "3 × ?"
  • rightExpression (string) — fixed side, e.g. "12"
  • answer (string) — value that replaces ?

--- build_number ---
Game: child builds the base number from part cards; tutor sets base and how many parts.
Per round REQUIRED fields:
  • baseNumber (number) — target whole number
  • partsCount (number) — 2, 3, or 4 (how many part slots)
Optional: readyFlag (boolean) — usually false at generation time

--- timer_challenge ---
Game: three quick problems A/B/C with answers under a time limit.
Per round REQUIRED fields:
  • timerExampleA, timerExampleB, timerExampleC (strings) — problem text
  • timerAnswerA, timerAnswerB, timerAnswerC (strings) — correct answers
  • timerSeconds (number) — seconds allowed, often 30

--- symbol_calc ---
Game: A, B, C shown as values; expression uses letters; child computes final value.
Per round REQUIRED fields:
  • symbolA, symbolB, symbolC (numbers)
  • symbolExpression (string) — uses A, B, C and operators, e.g. "A × B + C"
  • answer (string) — numeric result matching the expression

--- find_unknown ---
Game: A and B known; equation involves C; child solves for C.
Per round REQUIRED fields:
  • unknownA, unknownB (numbers)
  • unknownEquation (string) — must mention C, e.g. "A × B + C = 30"
  • answer (string) — numeric value of C

GENERAL RULES FOR ALL MECHANICS:
  • titleText is always the centered round/stage title shown to the child.
  • Use English only in all strings.
  • Numbers in prompts/answers must match age and difficulty rules above.
  • Do not invent extra keys for a mechanic; do not omit required keys.

EXAMPLE DESIGN PRINCIPLES
════════════════════════════════════════

Round progression within every stage (always follow this curve):

  Round 1 — simplest possible. Child understands the mechanic.
  Round 2 — same structure, slightly bigger or new numbers.
  Round 3 — add one new variable (more zones, new object type).
  Round 4 — introduce a twist, partial fill, or unusual angle.
  Round 5 — hardest round: inverse task, unexpected combination,
             or the concept flipped.

Every stage must include at least one round where the child
must think in reverse — e.g. "? × 3 = 12" not just "4 × 3 = ?"

Wrong answer options (for multi_choice, corridor_choice):
  Always use typical mistakes for that age and topic.
  Never use random numbers far from the correct answer.
  e.g. for "4 × 3 = ?" wrong options should be 10, 13 — not 42.

If tutor_prompt says "not just numbers" or "word problems":
  Wrap examples in short real-world context.
  e.g. instead of "4 × 3 = ?" write
  "3 monkeys each have 4 bananas. How many bananas total?"
  Keep language simple, age-appropriate, max 1 sentence.

Math topic coherence:
  All 6 stages must address the tutor_prompt topic.
  Do not wander into unrelated math.
  If tutor says "multiplication" — every stage practices
  multiplication in a different way across all 6 stages.

════════════════════════════════════════
ROUND DIVERSITY — STRICT (within each stage)
════════════════════════════════════════

Across the 5 rounds in ONE stage you must NOT repeat the same exercise.
Every round must differ clearly:

  • Different numbers / quantities (never copy-paste the same prompt text twice).
  • Round 1 is easiest; each next round increases difficulty or adds a twist.
  • At least two rounds must change problem structure (not only swap operands).
  • Wrong answers and distractors must change between rounds.

Forbidden: five rounds that are the same template with only one number changed.
